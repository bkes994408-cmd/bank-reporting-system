# MVP-4：根據使用者回饋或市場新需求，分析並規劃新的功能迭代

## 背景與目標
本文件作為 MVP-4 最後一項工作交付：
> 「根據使用者回饋或市場新需求，分析並規劃新的功能迭代」。

目標是在不破壞既有 MVP 穩定性的前提下，提出可在下一輪（MVP-5）落地的功能方案，並定義驗收標準與實施順序。

---

## 回饋來源與市場訊號（彙整）

### 使用者回饋（來自現行流程觀察）
1. **申報後追蹤成本高**：使用者需反覆查詢申報結果，缺少主動通知機制。
2. **錯誤修復迴圈長**：申報失敗時，無明確錯誤分類與建議，重送成本高。
3. **權限切換易混淆**：現行以 header 模擬角色便於 QA，但正式環境風險較高。
4. **管理端批次操作效率不足**：大量帳號調整或檢視時，步驟偏多。

### 市場/合規需求（金融申報場景常見）
1. **審計可追溯性**：操作軌跡與責任歸屬需更完整（who/when/what）。
2. **可靠性要求提升**：外部服務短暫異常時，系統需有重試/降級策略。
3. **最小權限與可治理**：RBAC 需從測試型機制過渡到正式 token claim。
4. **營運可觀測性**：需要更明確 SLA/SLO 與慢查詢/失敗趨勢可視化。

---

## 功能迭代候選（Backlog）

### P0（建議 MVP-5 優先）

#### 1) 申報狀態主動通知（Notification Center）
- **痛點對應**：申報結果需手動輪詢。
- **提案**：
  - 後端新增申報狀態變更事件（accepted/rejected/retrying/final-failed）。
  - 提供 webhook 與站內通知 API（二擇一先落地：站內通知）。
  - 前端新增「最近申報動態」區塊。
- **驗收標準**：
  - 申報狀態異動 5 秒內可查到通知紀錄。
  - 至少支援成功/失敗兩種通知模板。

#### 2) 申報失敗診斷與重送助手
- **痛點對應**：錯誤資訊難懂，修復成本高。
- **提案**：
  - 建立錯誤碼 taxonomy（憑證、格式、網路、上游異常）。
  - 失敗結果回傳可操作建議（next action）。
  - 支援「修正後重送」的一鍵帶入前次內容。
- **驗收標準**：
  - Top 10 常見錯誤皆有分類與建議。
  - 重送流程步驟減少至少 30%。

#### 3) RBAC 正式化（Token Claims）
- **痛點對應**：header 模擬角色不適合作為正式授權基礎。
- **提案**：
  - 將角色來源改為簽章 token claims。
  - 保留 header 模擬僅限 dev/test profile。
  - 為所有 protected routes 補齊整合測試。
- **驗收標準**：
  - production 設定下，header role 無效。
  - admin/operator route 皆有 allow/deny 測試覆蓋。

### P1（MVP-5 後半）

#### 4) 管理後台批次操作（Bulk Admin Actions）
- **痛點對應**：大量帳號管理效率不足。
- **提案**：
  - 新增批次角色調整（例如多選 user -> 指派 operator）。
  - 提供操作預覽與可回滾（短時窗）。
- **驗收標準**：
  - 100 筆帳號角色異動可於單次流程完成。
  - 產生完整 audit log。

#### 5) 可觀測性儀表板（SLO/SLA）
- **痛點對應**：難以即時判斷服務健康與風險。
- **提案**：
  - 擴充 `/api/metrics` 至成功率、P95 latency、外呼失敗率。
  - 提供 top N 慢端點與錯誤熱點。
- **驗收標準**：
  - 可在單一頁面看到日/週趨勢。
  - 可設定告警門檻（例如連續 5 分鐘失敗率 > 3%）。

## 建議執行節奏（兩個 Sprint）

### Sprint A（2 週）
- RBAC Token Claims 正式化
- 申報失敗診斷與錯誤分類 v1
- 申報狀態通知（站內通知版）

### Sprint B（2 週）
- 批次管理操作
- 可觀測性儀表板與告警規則
- 依 Sprint A 數據做第二輪調整

---

## 交付與衡量指標（KPI）
1. 申報失敗後平均修復時間（MTTR）下降 30%
2. 人工查詢申報結果次數下降 40%
3. 權限誤用事件（未授權存取）下降至 0
4. API P95 latency 維持在基準線 +10% 內
